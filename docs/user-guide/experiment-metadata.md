# experiment-metadata.yml

`experiment-metadata.yml` is the central artifact of a FACTS v2 experiment. It is generated by `setup-new-experiment`, edited by you, and then consumed by `generate-compose` to produce the Docker Compose file.

The file has two sections with different levels of user involvement:

- **[Top section](#top-section-your-main-editing-area)** — experiment parameters, input data paths, the module manifest, and workflows. This is what you fill in and own. Most of the required editing happens here.
- **[Module-specific section](#module-specific-section-review-dont-restructure)** — per-module inputs, options, and outputs. Pre-populated from each module's YAML and defaults files. You may need to adjust individual values (especially input file names), but you should not add, remove, or rename keys.

---

## Top section — your main editing area

<div class="annotate" markdown="1">
``` yaml
experiment_name: toy_experiment # (1)!

pipeline-id: aaa # (2)!
scenario: ssp585 # (3)!
baseyear: 2005 # (4)!
pyear_start: 2020 # (5)!
pyear_end: 2100
pyear_step: 10
nsamps: 1000 # (6)!
seed: 1234 # (7)!

fingerprint-dir: FPRINT # (8)!
location-file: location.lst # (9)!

climate_step: fair-temperature # (10)!
sealevel_modules: # (11)!
  - bamber19-icesheets
  - tlm-sterodynamics
framework_modules:
  - facts-total
esl_modules:
  - extremesealevel-pointsoverthreshold

workflows: # (12)!
  wf1: "bamber19-icesheets,tlm-sterodynamics"

module-specific-input-data: /path/to/your/module-specific-inputs # (13)!
general-input-data: /path/to/your/general-inputs # (14)!

output-data-location: ./experiments/toy_experiment/data/output # (15)!
```
</div>

1. Set by `--experiment-name` when you ran `setup-new-experiment`. Also the name of the directory under `experiments/`. You generally won't change this after setup.
2. A unique identifier for this pipeline run, passed as a command-line argument to every module. Set via `--pipeline-id` or filled in here before running `generate-compose`.
3. The emissions scenario (e.g. `ssp585`, `ssp245`). Passed to every module — some modules apply a name transform internally if their CLI expects a different convention.
4. The base year for sea-level projections. Shared across all modules.
5. Start, end, and step year for projection outputs. All three are passed to every module that produces time-series output.
6. Number of Monte Carlo samples. Passed to every module.
7. Random seed for reproducibility. Passed to every module that uses sampling.
8. Name of the GRD fingerprint data directory within your `general-input-data` path. Passed to any module that produces spatially-resolved (local) sea-level output.
9. Name of the location list file within your `general-input-data` path. Lists the lat/lon points for which local sea-level projections are computed.
10. The climate step module. Set by `--climate-step`. If you provided `--climate-step-data` instead, this field will be `null` and an `alternate_climate_data` field will hold the path you supplied. If you provided `--supplied-totaled-sealevel-data`, this field will also be `null`.
11. The full list of sea-level modules in this experiment. Set by `--sealevel-step`. `generate-compose` creates one Docker service per module listed here. If you provided `--supplied-totaled-sealevel-data`, this field will be `null` and a `supplied_totaled_sealevel_data` field will hold the path.
12. Each workflow is a named combination of sea-level modules whose outputs are summed by `facts-total` to produce a total sea-level projection. Entered interactively during `setup-new-experiment` when `facts-total` is the framework module. Add or edit workflows here if you want to change the combinations after setup.
13. **Required — fill this in before running `generate-compose`.** Absolute path to the directory containing your module-specific input data. Should have one subdirectory per module, named after the module (e.g. `bamber19-icesheets/`).
14. **Required — fill this in before running `generate-compose`.** Absolute path to the directory containing your general input data — `location.lst` and the GRD fingerprint directory live here.
15. Auto-set relative to your experiment directory. You can change this to write outputs elsewhere, but it must be an absolute or project-relative path that the Docker containers can reach.

## Module-specific section — review, don't restructure

!!! warning "Don't add, remove, or rename keys in this section"
    The structure of each module block is generated from the module's YAML configuration file and is read back by `generate-compose` by key name. Adding or removing keys, or renaming them (e.g. changing `climate_data_file` to `climate-file`), will cause `generate-compose` to fail or silently misroute arguments.

    What you **can** safely do: change the **values** of `inputs` and `options` fields to match your actual data files and preferred parameter settings.

Each module listed in the manifest gets its own block. The values are pre-populated from the module's `defaults_*.yml` — review them before running `generate-compose`.

<div class="annotate" markdown="1">
``` yaml
fair-temperature: # (1)!
  inputs:
    rcmip_fname: "rcmip/rcmip-emissions-annual-means-v5-1-0.csv" # (2)!
    param_fname: "parameters/fair_ar6_climate_params_v4.0.nc"
    cyear_start: 1850
    cyear_end: 1900
    smooth_win: 19
  options: # (3)!
    # Options inherited from top-level: pipeline-id, nsamps, seed, scenario
  image: "ghcr.io/fact-sealevel/fair-temperature:0.2.1" # (4)!
  outputs: # (5)!
    output-climate-file:
      value: "fair-temperature/climate.nc"
      output_type: global
    output-oceantemp-file:
      value: "fair-temperature/oceantemp.nc"
      output_type: global

bamber19-icesheets:
  inputs:
    climate_data_file: "fair-temperature/climate.nc" # (6)!
    slr_proj_mat_file: SLRProjections190726core_SEJ_full.mat # (7)!
    fingerprint_dir: # (8)!
    location_file:
  options:
    # Options inherited from top-level: pipeline-id, nsamps, rngseed, scenario, pyear-start, pyear-end, pyear-step, baseyear
    replace: 1 # (9)!
    chunksize: 50
  image: "ghcr.io/fact-sealevel/bamber19-icesheets:0.1.0"
  outputs:
    output-AIS-gslr-file:
      value: "bamber19-icesheets/AIS-gslr.nc"
      output_type: global # (10)!
    output-AIS-lslr-file:
      value: "bamber19-icesheets/AIS-lslr.nc"
      output_type: local # (11)!

# ... remaining modules follow the same pattern
```
</div>

1. One block per module in the manifest, keyed by module name in kebab-case. Generated by `setup-new-experiment` from the module's YAML and defaults files.
2. Input file paths are relative to the module's subdirectory within `module-specific-input-data`. Default values come from `defaults_*.yml` — update if your files have different names.
3. The comment lists which top-level parameters this module inherits automatically. You do **not** need to re-specify them here — `generate-compose` pulls them directly from the top section.
4. The container image for this module. Do not change this unless you are intentionally pinning to a different version.
5. Output paths are auto-generated relative to `output-data-location`. They wire outputs from one module as inputs to the next (e.g. `fair-temperature/climate.nc` becomes `climate_data_file` for downstream modules). Do not modify these.
6. Automatically set to reference the climate step's output. If you used `--climate-step-data`, this is automatically populated with the path you supplied. If you are editing the metadata manually, update this to your climate file path.
7. A module-specific input file pre-filled from `defaults_bamber19_icesheets.yml`. Verify this filename matches the file in your `module-specific-input-data/bamber19-icesheets/` directory.
8. Left blank intentionally — resolved at compose-generation time from the top-level `fingerprint-dir` and `location-file` fields. Leave empty.
9. Module-specific options pre-filled from the defaults file. You can change values (e.g. adjust `chunksize` for performance) but keep the keys exactly as-is.
10. `output_type: global` — a spatially-aggregated sea-level projection. `facts-total` uses this to identify outputs to sum across modules in a workflow.
11. `output_type: local` — a spatially-resolved projection at the locations in your `location-file`.
